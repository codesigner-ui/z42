# 代码组织规范

## 目录 README.md

**每个功能目录必须有一个 `README.md`，描述该目录的职责、核心模块和入口点。**

适用范围：`src/` 下最多 **3 层**目录（含 `src/` 本身算第 0 层，则子目录最深到第 3 层）。

```
src/                          ← 第 1 层，不需要（已有根 CLAUDE.md）
src/compiler/                 ← 第 2 层 ✓ 需要 README
src/compiler/z42c.semantics/  ← 第 3 层 ✓ 需要 README
src/compiler/z42c.semantics/src/  ← 第 4 层 ✗ 不自动加 README
```

第 3 层的 README 应在"核心文件"或"子目录"表中覆盖其子目录的职责，不依赖子目录的 README。

### README.md 模板（2026-07-01 起，5 段制）

README 是**人 review AI 实现的落点**——人不读源码，靠 README 判断"有啥、怎么验、关联谁"。
以下五段中 **① 职责 / ② 功能索引 / ③ 如何测试验证 / ④ 关联文档** 为必备，⑤ 核心文件强烈建议。

```markdown
# <目录名>

## 职责
[1–3 句：这个目录做什么，不做什么]

## 功能索引
[特性 / 能力 → 承载文件或入口的映射；人扫这里就知道"这里实现了什么"]
| 功能 | 入口 / 文件 |
|------|-----------|
| [能力一句话] | `Foo.z42` 的 `Bar()` |

## 如何测试验证
[精确到可复制的命令——AI 照着跑、人照着验；指明测本目录跑哪个 stage / 哪些用例]
    z42 xtask.zpkg test <stage>       # 覆盖本目录的测试（实际写成 bash 代码块）

## 关联文档
- 设计/机制：[book 章节](../../docs/book/src/...)
- 引入/演进：change `<name>`（`docs/spec/changes/` 或已归档）—— 需求↔迭代可追溯

## 核心文件
| 文件 | 职责 |
|------|------|
| `foo.rs` / `Foo.z42` | [一句话] |
```

> **旧模板**（职责/核心文件/入口点/依赖关系）已铺在 80+ 目录，逐步按本 5 段制补齐
> ②③④ 三段；新建目录直接用 5 段制。"入口点""依赖关系"可并入"功能索引""关联文档"或保留。

### 阅读顺序规则

**Claude 在阅读某个目录的代码前，必须先读取该目录的 `README.md`（如果存在），再决定读哪些具体文件。**

这样可以：
- 避免盲目读取无关文件浪费 token
- 快速定位关键入口，减少探索成本

### README 同步规则

**每次迭代完成后，Claude 必须检查所修改目录的 `README.md` 是否需要更新。**

触发更新的情形：
- 新增或删除了文件（核心文件表需同步）
- 改变了模块的对外入口点或公开 API
- 改变了依赖关系（新增或移除了对其他模块的依赖）
- 改变了目录的职责范围

**不需要更新的情形**：纯内部实现调整（如函数提取、变量重命名）且不影响上述四项。

> README 同步属于文档同步要求的一部分，归档前必须完成。未同步 = 未完成。

## 代码规模限制

### 文件行数

| 类型 | 软限制（建议拆分）| 硬限制（必须拆分）|
|------|-----------------|-----------------|
| Rust `.rs` 文件 | 300 行 | 500 行 |
| z42 `.z42` 文件 | 300 行 | 500 行 |

> 测试文件（`*_tests.rs`、`tests/<name>/` 下的 `.z42`）不计入限制，但单个测试文件超过 600 行时也应按功能拆分。

### 函数 / 方法行数

| 限制类型 | 行数 |
|----------|------|
| 软限制（超出需要重新审视） | 40 行 |
| 硬限制（必须提取子函数） | 60 行 |

> 计算方式：函数体内的非空、非注释行数。

### 类型（struct / class / impl block）行数

| 类型 | 硬限制 |
|------|--------|
| Rust `impl` 块 | 200 行（超出拆分为多个 impl 块或子模块） |
| z42 类型（class / struct / record） | 200 行（超出提取辅助类型） |

> **类型拆多文件时累计计入**：若某语言支持把一个类型拆到多个文件（如 partial），
> 按"类型整体"判定 200 行硬限——把归属同一类型的成员体积（除模块 / 命名空间 / 类壳
> 包裹之外的部分）累加，防止"类型超 200 行"被散落到多文件来绕过。
>
> 单文件 500 行硬限仍独立适用——多文件拆分主要解决"文件超限"，**不**解决
> "类型超限"。两条限制分别约束。

### 执行方式

- Claude 每次新增代码时，**主动检查**所在文件是否超出软限制
- 超出软限制时，**建议**拆分方案但不强制打断当前任务
- 超出硬限制时，**必须**在本次迭代内拆分，不得推迟
- 拆分是独立的 `refactor` 变更，单独 commit，不与功能变更混合
