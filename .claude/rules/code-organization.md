# 代码组织规范

## 目录 README.md

**每个功能目录必须有一个 `README.md`，描述该目录的职责、核心模块和入口点。**

适用范围：`src/` 下最多 **3 层**目录（含 `src/` 本身算第 0 层，则子目录最深到第 3 层）。

```
src/                          ← 第 1 层，不需要（已有根 CLAUDE.md）
src/compiler/                 ← 第 2 层 ✓ 需要 README
src/compiler/z42.Compiler/    ← 第 3 层 ✓ 需要 README
src/compiler/z42.Compiler/Codegen/  ← 第 4 层 ✗ 不自动加 README
```

第 3 层的 README 应在"核心文件"或"子目录"表中覆盖其子目录的职责，不依赖子目录的 README。

### README.md 模板

```markdown
# <目录名>

## 职责
[1–3 句：这个目录做什么，不做什么]

## 核心文件
| 文件 | 职责 |
|------|------|
| `foo.rs` / `Foo.cs` | [一句话] |

## 入口点
[主要对外暴露的类型/函数/模块]

## 依赖关系
[依赖哪些上层/下层模块，可选]
```

### 阅读顺序规则

**Claude 在阅读某个目录的代码前，必须先读取该目录的 `README.md`（如果存在），再决定读哪些具体文件。**

这样可以：
- 避免盲目读取无关文件浪费 token
- 快速定位关键入口，减少探索成本

### README 同步规则

**每次迭代完成后，Claude 必须检查所修改目录的 `README.md` 是否需要更新。**

触发更新的情形：
- 新增或删除了文件（核心文件表需同步）
- 改变了模块的对外入口点或公开 API
- 改变了依赖关系（新增或移除了对其他模块的依赖）
- 改变了目录的职责范围

**不需要更新的情形**：纯内部实现调整（如函数提取、变量重命名）且不影响上述四项。

> README 同步属于文档同步要求的一部分，归档前必须完成。未同步 = 未完成。

## 代码规模限制

### 文件行数

| 类型 | 软限制（建议拆分）| 硬限制（必须拆分）|
|------|-----------------|-----------------|
| Rust `.rs` 文件 | 300 行 | 500 行 |
| C# `.cs` 文件 | 300 行 | 500 行 |

> 测试文件（`*_tests.rs`、`*Tests.cs`）不计入限制，但单个测试文件超过 600 行时也应按功能拆分。

### 函数 / 方法行数

| 限制类型 | 行数 |
|----------|------|
| 软限制（超出需要重新审视） | 40 行 |
| 硬限制（必须提取子函数） | 60 行 |

> 计算方式：函数体内的非空、非注释行数。

### 类型（struct / class / impl block）行数

| 类型 | 硬限制 |
|------|--------|
| Rust `impl` 块 | 200 行（超出拆分为多个 impl 块或子模块） |
| C# `class` / `record` | 200 行（超出提取辅助类型） |

### 执行方式

- Claude 每次新增代码时，**主动检查**所在文件是否超出软限制
- 超出软限制时，**建议**拆分方案但不强制打断当前任务
- 超出硬限制时，**必须**在本次迭代内拆分，不得推迟
- 拆分是独立的 `refactor` 变更，单独 commit，不与功能变更混合

## 规范冲突检测

**Claude 在实施任何变更时，若发现规范文档之间存在冲突、不一致或冗余，必须立即：**

1. **停下来指出冲突**，描述：哪两个文档、哪条规则、冲突点是什么
2. **提出建议的解决方案**（以哪个为准，或如何合并）
3. **等待 User 裁决**后再继续实施
4. **裁决结果同步到所有相关文档**，确保唯一真相来源

> 规范冲突属于"规范偏差"，优先级高于当前任务推进。
