## 一、目标结构（end state）

重构完成后，文档收敛为**三个 docs 子目录 + 变更工作区 + 目录 README + 两个瘦入口**：

```
docs/
├── book/        【知识库·唯一SoT】mdBook（纯 markdown）——系统"是什么"：语法/机制/架构/权衡
├── workflow/    【操作手册】build / test / ci / release / debug 的具体命令
└── agent/       【面向大模型的协作材料】
    └── rules/     开发规范（流程/风格/提交…；后续可在 agent/ 下新建其它分类子目录）

docs/spec/changes/     【变更工作区】进行中 change + ACTIVE.md 账本 —— 保留
src/**/README.md       【目录说明】5 段制 —— 人 review 的落点
CLAUDE.md / AGENTS.md  【瘦入口】让任意大模型 import 上面的文件 —— 最后一步才配
```

> `docs/design/` **不在目标结构里**——它是待迁移的旧仓库，迁完即删（见第三节 D1/D2）。

## 二、四类文档职责（按生命周期切分）

混乱的根因是"知识 / 变更 / 规范 / 目录说明"挤在一起且互相过时。按生命周期彻底分开：

| 类别 | 是什么 | 生命周期 | 权威位置 | 受众 |
|------|--------|---------|---------|------|
| **① 知识库** | 系统当前**是什么** | 永久·永远最新 | `docs/book/` | 用户 + 维护者 + 大模型 |
| **② 变更工作区** | 一次迭代**在做什么** | 短命·做完即弃 | `docs/spec/changes/` | 大模型 + 审批的 User |
| **③ 开发规范** | 大模型/人**怎么干活** | 稳定·少变 | `docs/agent/rules/` | 大模型 + 协作者 |
| **④ 目录说明** | 某目录**有啥/怎么测/关联谁** | 随代码变 | 各 `src/**/README.md` | 人 review 入口 |

附：**操作手册**（`docs/workflow/`，怎么操作的命令）独立于以上四类。

**SoT 铁律**：每条知识只有一个权威位置，其他地方只能链接，不得复制。发现同一事实两处各写一份
→ 按[规范冲突检测](../CLAUDE.md)停下，留一处、另一处改链接。

## 三、关键决策记录

| ID | 决策 | 理由 |
|----|------|------|
| **D1** | **不保留 design 作"基础层"**；"基础 vs 详细"竖切进 book（概览页=基础，深层章节=详细），不横切成两个目录 | 两份描述同一件事 → 必然漂移 = 我们要逃离的原病 |
| **D2** | `docs/design/` 定位为**待迁移旧仓库**：不再往里写新东西；新知识一律进 book；全部迁完+校对后**删除** design/ | 单一权威、消除冗余 |
| **D3** | "基础说明"= book 各部分**概览页**（`book/src/<part>/README.md`）；零散小主题 → `book/appendix/` | book 自身分层已覆盖两种深度 |
| **D4** | 正式语言规范（normative reference，Rust *Reference* 那种）**推迟到 1.0**，届时从 book 派生；现在不养两套 | pre-1.0 两三人维护，养两套必烂一套 |
| **D5** | book 用 **mdBook**（纯 markdown，`mdbook serve` 可渲染发布） | 与 Rust 技术栈一致，无额外负担 |
| **D6** | 开发规范**模型中立**，居 `docs/agent/rules/`；`CLAUDE.md`/`AGENTS.md` 仅瘦入口指进去，**最后一步**才配 | 文档为所有大模型设计，不硬绑 Claude 专属机制 |
| **D7** | `docs/spec/archive/`（590 历史目录）**暂留**，book 全面覆盖后单独清理 | 死历史 git 已留痕，不阻塞当前 |
| **D8** | 目录 README **5 段制**：职责 / 功能索引 / 如何测试验证 / 关联文档 / 核心文件 | 人不读源码，靠 README 快速 review（模板见 [code-organization.md](code-organization.md)） |

## 四、知识「上浮」规则（本体系不过时的保证）

**每个 change 归档（[workflow.md 阶段 9](workflow.md)）时，凡改变了语言/编译器/VM/stdlib 的对外
行为或内部机制，其知识必须上浮进 `docs/book/` 对应章节——然后变更目录随后可删（git 留痕）。**

以前 design（知识）与 spec（变更）并列，知识散落在数百个历史 change 里没人捞 → design 过时、
archive 堆到 590。用"上浮"把知识收敛到 book 单一权威，是 design 能被替换清理的前提。

## 六、语种策略（沿用）

对外文档（用户/贡献者）英文；内部文档（开发流程/实现细节）中文。book 面向用户与维护者双受众，
**默认中文正文、关键术语保留英文**，成熟章节再评估是否出英文版。
