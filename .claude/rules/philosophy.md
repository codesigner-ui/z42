# 实现哲学与设计完整性

> 流程主线（阶段 / Scope / commit / 归档）见 [workflow.md](workflow.md)。
> 这份文件沉淀**做选择时的判断准则**：何时停、何时改设计、何时记延后。

---

## 实现方案原则（必须遵守）

**优先采用最终的、架构上最合理的方案，不得擅自选择临时方案（quick fix、hack、不完整实现）。**

- 如果存在"临时方案"与"最终方案"之分，默认实施最终方案
- 若当前条件确实迫使必须先采用临时方案（如依赖未就绪、工作量超出当前 Scope），**必须先明确说明原因，并得到 User 确认后方可实施**
- 不得以"先实现临时版本，以后再重构"为由跳过最终设计

### 不为破坏性顾虑而牺牲最佳方案（2026-05-14 强化）

**z42 pre-1.0 阶段，"破坏性"不是阻挡最佳方案的理由。**

- 跨 spec / 跨模块改动时，若发现既有 API / 命名 / 文件位置等不是最佳选择，**直接改成最佳方案**——不要为"会破坏 N 个调用点"而退而求其次
- 调用点的更新是机械工作；调用点数量不影响"何为最佳方案"的判断
- 唯一约束：改完之后整套测试必须 GREEN（修好所有破坏的调用点是同一次迭代的工作）

**典型场景**：
- 加新 stdlib 类时发现现有命名空间组织不合理 → 调整组织（移文件、改 namespace）
- 设计一致性要求老 API 改名 → 改名（连带所有调用方一起改）
- 一个类的字段顺序 / 类型在跨平台分支下不一致 → 统一（多平台 Rust 改动同时进）

> 与 [`不为旧版本提供兼容`](#不为旧版本提供兼容2026-04-26-强化) 配合：兼容规则说"已发布的旧产物不背"，本规则说"当前 in-flight 的破坏面也不背"——两条合起来意思是：最佳方案就是最佳方案，破坏面用同一次迭代清零。

### 修复必须从根因出发（2026-04-26 强化）

**修 Bug / 限制时，先识别根因（root cause），然后从源头修复，禁止在症状层面打补丁。**

典型反例（即"症状级临时方案"）：

- ❌ 在比较 / 转换层加同名兼容分支（`if (sourceName == targetName) return true`）来掩盖"类型对象本不应是不同种类"的设计错误
- ❌ 在调用方 try/catch 吞掉异常以"修好" 测试，而不去问 callee 为什么抛
- ❌ 用全局 if-else 特例分支处理某个具体类型，绕过通用机制
- ❌ 在解析失败时降级为 sentinel 值（`Z42PrimType("Unknown")` / 空字符串），让下游用启发式去"猜"

**根因修复检查清单**：

1. **数据为什么会到达错误状态？** —— 顺着调用栈反推产出该状态的代码点
2. **该代码点为什么写成这样？** —— 是不是因为依赖未就绪 / 信息不全 / 设计妥协？
3. **修复时改的是产出端还是消费端？** —— 必须改产出端（让数据从源头就正确），消费端的兼容分支只是症状级补丁
4. **修复后该错误状态是否物理上不可能再发生？** —— 是 → 根因修复；否（仍可能发生但被绕过） → 临时方案

**典型场景：跨阶段类型 / 元数据降级**

当某 Phase 1 因上下文不全把 X 降级为 Y（占位类型 / sentinel），Phase 2 信息齐了再用 X 时，正确做法是 **Phase 2 加一个 fixup pass 把 Y 升级回 X**（一次性，物理消除降级状态），而不是让所有消费端都加 `if (Y) treat-as-X` 的容错逻辑。

> 参见经典做法：C# / Java 编译器的"两阶段类型加载"（先建骨架 → 再填字段），消除 self-reference / forward-reference 时的降级。

### 系统性修复 vs Scope 控制

"系统性修复"不等于"Scope 蔓延"。判断标准：

- ✅ **属于本变更 Scope 内的根因**：修源头，禁止打补丁
- ⚠️ **根因在本变更 Scope 之外**：先停下报告 User，描述根因 + 请求 Scope 扩展（参见 W1 / VM-zpkg 案例的 Scope 扩展处理），User 批准后从根因修

**不允许"因为 Scope 限制就在症状层面修复"** —— 那只会在远端留下技术债。如果 Scope 真的不允许，留 backlog 给独立变更彻底修，本变更对该问题不动。

### 不为旧版本提供兼容（2026-04-26 强化）

**z42 当前处于快速迭代阶段（pre-1.0），不考虑旧版本兼容。** 任何格式 / API /
语义变更直接以最简形式 emit / 实施，**不留兼容路径**。

适用范围：

- **zbc / zpkg 二进制格式** — 字段添加 / 移除直接做；version bump 后旧版本
  zbc **不可读**（解码器无需 fallback）。残留旧产物用 `z42 xtask.zpkg regen`
  重生
- **stdlib 公开 API** — 接口 / 类 / 方法删除或重命名直接做；不留 alias / deprecation
- **IR 指令** — 新增字段 / 改语义直接做；旧字段不保留
- **TypeChecker / 编译器内部约定** — 同上
- **工程文件 / CLI 命令** — 同上

**禁止行为：**

- ❌ "保留旧字段以兼容已编译的 .zbc" — 用户重新 regen 即可
- ❌ "deprecation period 期内同时支持新旧 API" — 直接换
- ❌ "旧 minor version zbc 兼容路径" — minor / major 都不兼容旧版
- ❌ "灰度迁移，新旧并存" — 一次切干净

**唯一例外**：与外部世界的接口（操作系统 syscall / 文件格式如 UTF-8 等
通用规范）— 这些不是 z42 本身的"旧版本"。

**节奏说明**：到 z42 进入 1.0 / 自举完成阶段后，本规则会重写为"语义版本 +
deprecation 周期"。当前处于探索期，**任何兼容性投入都是过早优化**。

> 写代码 / spec 时主动检查：是否有"为兼容而保留"的逻辑分支？有 → 删除。

> 具体 `.zbc` / `.zpkg` 格式 version bump 的协调 checklist（哪些文件必须同步改 + commit 前自检命令）见 [version-bumping.md](version-bumping.md)。

## 设计完整性原则（必须遵守）

**代码必须时刻保持设计精良、可维护、易扩展。当现有设计无法干净承载新需求时，立即停下来与 User 讨论，禁止打补丁绕过。**

### 触发条件

阶段 1（探索）/ 阶段 5（design.md）/ 阶段 7（实施）任一节点，发现以下信号 → 停：

- 需要在通用机制里加 special case 才能塞下当前需求
- 需要绕过现有抽象 / 封装 / 分层才能实现
- 现有数据结构无法表达新需求，需要塞 sentinel / Optional / 类型 union
- 修改某个类时发现它已经承担了 3+ 不相关职责
- 实施时 spec 描述的简洁行为对应的代码却处处充满分支

### 必须做的动作

向 User 提出两个选项 + 推荐：

1. **改进设计** — 重构、调整数据结构、抽象重新分层，让设计能优雅承载需求
2. **调整需求** — 重新审视需求边界，看是否本就不该这样要求

模板：

> 当前设计：<一句话描述>
> 新需求冲突点：<具体冲突>
> 选项 A（改设计）：<改动范围 + 代价>
> 选项 B（改需求）：<需求收窄方向 + 代价>
> 推荐：<X>，因为 <Y>

### 禁止行为

- ❌ 设计不匹配时硬塞实现（special case、if-else 分支、绕过封装、sentinel 值）
- ❌ "先这样实现以后再重构" — 不在 User 明确同意前提下走临时路径
- ❌ 顺手扩展某个抽象使其变得臃肿，只为塞下不属于该抽象的功能
- ❌ 批量授权模式下自行决定 — 属于 [workflow.md 阶段 6.5](workflow.md) 中断条件 4（决策点未覆盖）/ 8（架构性发现）

### 与其他规则的关系

- 与 **修复必须从根因出发**（本文件）配合：根因若是"设计已偏移"，按本规则停下
- 与 **规范冲突检测**（[CLAUDE.md](../CLAUDE.md)）配合：spec 与代码现状冲突时，本规则适用
- 与 **实现方案原则**（本文件）配合：本规则强调"何时停"，实现方案原则强调"停下后选最终方案"

## 延后特性管理（必须遵守）

**所有延后特性（无论设计期还是实施期识别）一律就近记录到对应 `docs/design/<dir>/<feature>.md` 的 "Deferred / Future Work" 段；`docs/roadmap.md` 的 "Deferred Backlog Index" 段统一索引指向这些条目。**

**记录格式（design doc Deferred 段中的单个条目）：**

```markdown
### <ID>: <标题>

- **来源**：spec 名 / review.md 段落 / 实施期 spec
- **触发原因**：为什么暂不做
- **前置依赖**：解锁需要什么
- **触发条件**：什么时候应该回来做
- **当前 workaround**（如有）：用户/调用方现在如何绕过
```

**判断标准（识别时机不再决定记录位置，但影响 ID 风格）：**

- **设计期延后** = 在写第一行实现代码前就决策的"暂不引入"；它是当前特性"完整设计图景"的一部分。命名建议 `<feature>-future-<short-name>`。
- **实施期延后** = 实施过程中浮现、经 User 裁决"暂不做"的债务。命名沿用历史 `D-N` 体系或 `<spec-name>-residual`。

**为什么集中到 design doc：**

- 单一权威：未来工作就近放在特性自己的设计文档里，读者看一份文档即可了解"该特性当前形态 + 未来空间"
- 便于跨 spec 追溯：`roadmap.md` 的 "Deferred Backlog Index" 表横向索引所有 design doc 的 Deferred 段，集中扫描成本不高于过去单独 deferred.md
- 避免决策分散：曾有 `docs/deferred.md` 单独债务簿，但实践中导致同一特性的设计期延后与实施期延后被分到两处，违反"单一真相来源"原则；2026-05-10 取消并合并

**禁止反例：**

- ❌ 把延后写进 `docs/deferred.md`（该文件已废弃，不存在）
- ❌ 同一项在多个 design doc 都登记（决策权威分散）
- ❌ 把"未来某个 phase 可能引入"的展望式描述写进 design doc Deferred 段（那属于 roadmap 或 features.md）
- ❌ 在 design doc Deferred 段写延后但没在 `roadmap.md` Deferred Backlog Index 加索引行（破坏集中追溯能力）

**与 feedback_problem_first_then_defer 的关系：**

- 该规则覆盖"实施期发现新问题"的处理流程（停 → 汇报 → User 裁决 → 进对应 design doc Deferred 段 + roadmap Deferred Backlog Index）
- 两条规则共同保证：所有延后决策有就近留痕 + 跨 spec 可追溯
